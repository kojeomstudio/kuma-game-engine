# 기술 검토: 학습용 게임 엔진 (C++ / C#)

> **목적**: 소프트웨어 렌더링부터 시작해 점진적으로 발전시킬 학습용 게임 엔진을 위해
> 언어, 그래픽스 API, 프레임워크/라이브러리 후보를 추려 기술적 검토.
>
> **작성일**: 2026-07-12  ·  **상태**: 검토 중

---

## 1. 요구사항 및 설계 목표

| 구분 | 내용 |
|------|------|
| **1차 목표** | 소프트웨어 렌더링(CPU 래스터라이저) 직접 구현 — 그래픽스 파이프라인의 원리 학습 |
| **2차 목표** | 하드웨어 가속(OpenGL/Vulkan/D3D12)으로 단계적 업그레이드 |
| **아키텍처** | **C++ 핵심 엔진 + C# 스크립팅 레이어** (P/Invoke 기반 C++/C# interop 학습) |
| **환경** | Windows 우선, 확장성 고려 |
| **학습 중심** | 렌더링 파이프라인, 메모리/성능, 언어 간 interop, 엔진 아키텍처 |

### 핵심 설계 원칙
- **학습 우선**: 매직(블랙박스)이 적고 내부가 투명한 도구 우선.
- **점진적 복잡도**: 소프트웨어 렌더링(창 + 픽셀 버퍼) → HAL → 고수준 엔진.
- **interop 학습 가치**: 실무에서 흔히 쓰이는 C++ 코어 + C# 스크립팅 구조 경험.

각 후보는 아래 기준으로 평가합니다.

6. **빌드/도구** — 설정 오버헤드가 적은가?

## 3. 후보군 검토

### 3.1 창/입력/플랫폼 추상화 (소프트웨어 렌더링용 호스트)

#### A. **SDL3** ⭐ 추천
- **현황**: 2025년 정식 릴리스. C 기반이라 C++/C# 모두에서 네이티브 호출 가능.
- **소프트웨어 렌더링**: `SDL_Surface` + `SDL_UpdateWindowSurface()` 또는 텍스처 업로드 경로가 매우 단순. CPU `FrameBuffer` → 화면 1:1 매핑이 학습 예제의 정석.
- **확장성**: SDL3의 신규 **GPU API**(D3D12/Vulkan/Metal 래퍼)로 자연스럽게 HW 가속 단계로 전환 가능.
- **interop**: C ABI → C# P/Invoke 가장 깔끔함. `SDL3-CS` 등 바인딩 존재.
- **학습 자료**: CppCon 2025(Mike Shah), 다수의 엔진 제작 튜토리얼.

#### B. **GLFW**
- 창/입력에 특화, 매우 미니멀. 소프트웨어 렌더링 시 OpenGL 컨텍스트 텍스처 업로드 방식 사용.
- 단점: 오디오/이미지 로딩 등은 외부 라이브러리 조립 필요. SDL3 대비 기능 범위 좁음.

#### C. **Win32 직접 호출**
- 가장 낮은 추상화 → "무엇이 일어나는지" 가장 투명 (학습 가치 높음).
- 단점: 플랫폼 종속, 코드 장황, 학습 자료 노후화. C# interop 시 직접 P/Invoke.

#### D. **Raylib**
- C 기반, 즉시 모드 API로 초보자 친화적. 소프트웨어 렌더링용 텍스처 블리트 지원.
- 단점: 엔진 프레임워크에 가까워 "엔진을 만드는" 학습 목표에는 너무 많은 것을 대신해줌.

### 3.2 그래픽스 API (HW 가속 단계 후보)

| API | 학습 가치 | 복잡도 | 비고 |
|-----|-----------|--------|------|
| **OpenGL** | 중 | 중 | 자료 방대. 단, 레거시/중단 추세. 학습용 1단계엔 무난. |
| **Vulkan** | 매우 높 | 매우 높 | 현대 GPU API. 학습 부담 크지만 파이프라인 이해에 최고. |
| **Direct3D 12** | 매우 높 | 매우 높 | Windows 환경에서 Vulkan과 대등. DXC/PIX 도구 강력. |
| **SDL3 GPU API** | 중-고 | 중 | Vulkan/D3D12/Metal 추상화. 진입장벽 완화. |
| **WebGPU (wgpu)** | 중 | 중-저 | 차세대 크로스플랫폼. 미래 지향적이나 생태계 성숙도 진행중. |

> **소프트웨어 렌더링 단계에서는 그래픽스 API 선택이 당장 필요하지 않음.**
> SDL3 호스트 위에서 CPU `FrameBuffer`를 텍스처/서피스로 띄우는 것부터 시작.

### 3.3 C++ / C# interop 방식

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **P/Invoke (C ABI)** ⭐ | C++ 코어를 `extern "C"`로 노출, C#에서 DllImport | 단순·표준·자료 풍부 | 수동 마셜링, 호출 오버헤드 |
| C++/CLI | Microsoft 확장으로 브릿지 | C++ 객체 직접 노출 | Windows/MSVC 전용, .NET Core 지원 부족 → **비추천** |
| **C++/C# (최신)** | .NET 9+ 개선된 interop, `[LibraryImport]` 소스 제너레이터 | 마셜링 최적화, AOT 친화 | 최신 문법 학습 필요 |
| SWIG / 바인딩 생성기 | 헤더 → C# 바인딩 자동 생성 | 대규모 API 자동화 | 도구 학습 곡선 |
| FlatBuffers / 공유 메모리 | 데이터 직렬화 대신 메모리 공유 | 초저지연 통신 | 설계 복잡, 오버킬 |

## 5. 권장 조합 (제안)

### 🎯 추천: **SDL3 + C++ 코어 + C# 스크립팅(P/Invoke)**

```
┌─────────────────────────────────────────────┐
│  C# Scripting Layer (게임 로직, 씬, 컴포넌트) │
│        via P/Invoke (extern "C" API)         │
├─────────────────────────────────────────────┤
│  C++ Engine Core                             │
│   • Software Renderer (CPU FrameBuffer)      │
│   • Math / Memory / Resource                 │
├─────────────────────────────────────────────┤
│  Platform Layer: SDL3 (창/입력/오디오/업로드) │
└─────────────────────────────────────────────┘
```

**선택 근거**
1. **소프트웨어 렌더링** — SDL3의 서피스/텍스처 업로드가 CPU 픽셀 버퍼 출력의 가장 깔끔한 경로.
2. **interop 학습** — SDL3은 순수 C ABI라 P/Invoke 경계가 명확하고, `[LibraryImport]` 등 최신 .NET interop 실습에 적합.
3. **확장성** — 동일한 SDL3 위에서 GPU API로 HW 가속 단계로 자연스럽게 이행 가능. API 전환 비용 최소.
4. **학습 자료** — SDL3 + 엔진 제작 튜토리얼이 2025년 기준으로 가장 풍부.

## 6. 학습 로드맵 (단계별)

| 단계 | 목표 | 산출물 |
|------|------|--------|
| **1. 소프트웨어 렌더러** | CPU `FrameBuffer`, 선/삼각형 래스터라이저, 텍스처 매핑, 깊이 버퍼 | SDL3 창에 CPU 렌더링 출력 |
| **2. 수학/자원 계층** | Vec/Mat, 쿼터니언, 메쉬/텍스처 로더 | 기하학 파이프라인 |
| **3. C# 스크립팅 바인딩** | `extern "C"` API 설계, P/Invoke, 씬/엔티티 노출 | C#에서 엔진 제어 |
| **4. HW 가속 (선택)** | SDL3 GPU API 또는 Vulkan/D3D12 | GPU 렌더 패스 |
| **5. 엔진 시스템** | ECS/씬 그래프, 에디터, 오디오 | 통합 엔진 |
