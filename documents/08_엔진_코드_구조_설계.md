# 엔진 코드 구조 설계 검토

> **목적**: 렌더러 기반의 가벼운 엔진 코드 구조를 설계한다.
> GLFW/glad를 어디에 배치하고, 계층을 어떻게 분리할지 검토.
> 학습 단계에 맞춰 "완벽한 추상화"가 아닌 "올바른 분리선"에서 시작.
>
> **관련**: `02_그래픽스_추상화_및_C샵_래퍼_검토.md` (RHI), `03_프로젝트_설정_OpenGL_시작.md` (폴더 구조)
>
> **작성일**: 2026-07-17

## 1. 설계 원칙 — "가볍되 올바른 분리선"

학습 단계에서 두 가지 위험을 피합니다:

| 위험 | 설명 | 대응 |
|------|------|------|
| **오버엔지니어링** | 처음부터 완벽한 RHI 다중 백엔드 추상화 시도 → 보일러플레이트에 묻혀 그래픽스 학습이 안 됨 | GL 백엔드 1개로 시작, 나중에 추상화 |
| **구조 없음** | main.cpp 한 파일에 다 때려넣기 → 나중에 RHI 분리 시 전면 재작성 | **처음부터 물리적 폴더 분리만** 유지 |

**핵심 철학**: 지금은 "추상 인터페이스"를 만들지 않고, **"폴더/파일 물리 분리"**만 해둔다. 추상화는 두 번째 백엔드(Vulkan/D3D)를 추가할 시점에 도입. 그때 분리된 코드를 인터페이스 뒤로 옮기면 됨.

## 2. 계층 아키텍처 (현대 엔진 컨센서스)

Hazel(TheCherno), Unreal, O3DE 등에서 검증된 모델:

```
┌─────────────────────────────────────────────┐
│  Application        (엔트리, 메인 루프)       │
├─────────────────────────────────────────────┤
│  Renderer           (고수준: 메시, 카메라)    │
├─────────────────────────────────────────────┤
│  RHI                (추상: 버퍼, 셰이더)      │
├─────────────────────────────────────────────┤
│  OpenGL Backend     ← glad가 여기에만 존재    │
│  (Vulkan Backend)   ← 향후 추가              │
├─────────────────────────────────────────────┤
│  Platform           (GLFW: 창, 입력)         │
└─────────────────────────────────────────────┘
```

### 2.1 핵심 규칙: 의존성 방향은 아래로만

- 상위 계층은 하위 계층을 **알 수 있지만**, 하위는 상위를 모른다.
- `glad/gl.h`와 `GLFW/glfw3.h`는 **특정 폴더(Platform/과 RHI/OpenGL/) 안에서만** `#include` 된다.
- 이 규칙 하나만 지켜도 나중에 백엔드 교체가 가능해진다.

### 2.2 glad의 위치 — 가장 중요한 결정 ⭐

**조사 결과 컨센서스**: glad는 **OpenGL 백엔드 구현체 안에만** 존재해야 한다.
(출처: [How do game engines implement multiple rendering platforms?](https://gamedev.stackexchange.com/questions/177756/), [O3DE Atom RHI](https://docs.o3de.org/docs/atom-guide/dev-guide/rhi/rhi/), [TheCherno Engine series](https://www.youtube.com/watch?v=HFyHIc89z1g))

```
잘못된 예: main.cpp, Renderer.cpp 어디서든 #include <glad/gl.h>
올바른 예: 오직 sources/Platform/GLFW/ 와 sources/RHI/OpenGL/ 내에서만 include
```

이렇게 하면, 나중에 Vulkan 백엔드를 추가할 때 glad 의존성이 전혀 없는 깨끗한 경계가 유지됨.

## 3. 권장 폴더 구조 (가벼운 시작)

이전 `03_문서`에서 제안한 구조를 구체화. **지금 당장 만들 폴더** (빈 폴더라도):

```
kuma-game-engine/sources/
├── Core/                       ← 엔진 뼈대
│   ├── Application.{h,cpp}     ← 메인 루프, 엔진 오너 (나중에 작성)
│   ├── Entry.h                 ← main() 진입 (나중에)
│   └── Logger.{h,cpp}          ← 로깅 (선택, spdlog 나중에)
│
├── Platform/                   ← OS/창/입력 — GLFW 의존
│   ├── Window.{h,cpp}          ← GLFW 창 래핑 (추상 인터페이스 없이, 구체 타입)
│   ├── Input.{h,cpp}           ← 키/마우스 입력 래핑
│   └── GLFW/                   ← GLFW 관련 세부 구현 (필요 시)
│
├── RHI/                        ← 그래픽스 추상화 (나중에 인터페이스 도입)
│   ├── Buffer.h                ← 버퍼 선언 (지금은 비워둬도 됨)
│   ├── Shader.h
│   ├── Texture.h
│   └── OpenGL/                 ← OpenGL 백엔드 — glad 의존
│       ├── GLContext.{h,cpp}   ← GL 컨텍스트 초기화 (gladLoadGL 호출)
│       ├── GLBuffer.{h,cpp}    ← glBufferData 등
│       └── GLShader.{h,cpp}
│
├── Renderer/                   ← 고수준 렌더링
│   ├── Camera.{h,cpp}          ← 카메라/뷰행렬
│   ├── Mesh.{h,cpp}            ← 메시 데이터
│   └── Renderer.{h,cpp}        ← 렌더 패스 오케스트레이션
│
├── Math/                       ← 수학 (직접 구현 or glm)
│   └── (Vec, Mat ...)
│
└── main.cpp                    ← 현재 위치. 나중에 Core/Entry.cpp로 이동
```

### 3.1 왜 지금 빈 폴더를 만드는가?
- CMake에서 폴더가 있으면 VS 솔루션 탐색기에 그대로 반영 → 시각적 구조 확립
- 처음부터 분리 습관 형성
- "이 코드는 어디에?" 질문에 폴더 구조가 답이 됨

### 3.2 지금 vs 나중에 작성할 것
| 폴더 | 지금 (학습 단계) | 나중에 (엔진화) |
|------|------------------|-----------------|
| `Platform/` | Window/Input (GLFW 래핑) | 인터페이스화, 다른 OS 지원 |
| `RHI/OpenGL/` | GLContext (glad 초기화만) | Buffer/Shader/Texture 추상화 |
| `RHI/` (루트) | 비어있음 | IRHIDevice 등 인터페이스 |
| `Renderer/` | Camera, 단순 Mesh | 렌더 패스, 머티리얼 시스템 |
| `Core/` | Application (메인 루프) | ECS, 씬 그래프 |

## 4. GLFW의 위치 — Platform 계층

GLFW는 "창 + 입력"을 담당하므로 **Platform 계층**에 속한다.

### 4.1 GLFW 래핑 패턴 (Window 클래스)
GLFW의 C API를 C++ 클래스로 감싸는 것이 첫 단계. **추상 인터페이스 없이** 구체 타입으로 시작.

```cpp
// sources/Platform/Window.h
#pragma once
struct GLFWwindow;            // 전방 선언 (헤더에서 glfw3.h 불러오지 않음 → 컴파일 속도 ↑)

namespace kuma {

class Window {
public:
    Window(int width, int height, const char* title);
    ~Window();

    bool shouldClose() const;
    void swapBuffers();
    void pollEvents();

    // 입력 접근 (나중에 Input 클래스로 분리)
    bool isKeyPressed(int key) const;

private:
    GLFWwindow* m_window = nullptr;
};

} // namespace kuma
```

```cpp
// sources/Platform/Window.cpp
#include "Window.h"
#define GLFW_INCLUDE_NONE
#include <GLFW/glfw3.h>        // GLFW는 .cpp 안에서만

namespace kuma {

Window::Window(int width, int height, const char* title) {
    glfwInit();
    glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 3);
    glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 3);
    glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
    m_window = glfwCreateWindow(width, height, title, nullptr, nullptr);
    glfwMakeContextCurrent(m_window);
}

Window::~Window() {
    if (m_window) glfwDestroyWindow(m_window);
    glfwTerminate();
}

bool Window::shouldClose() const { return glfwWindowShouldClose(m_window); }
void Window::swapBuffers()        { glfwSwapBuffers(m_window); }
void Window::pollEvents()         { glfwPollEvents(); }

} // namespace kuma
```

**핵심 포인트**:
- **헤더(`.h`)에는 GLFW 타입을 전방 선언만** (`struct GLFWwindow;`) — `glfw3.h`를 끌어오지 않음. 이게 의존성 누출을 막는 핵심 기법.
- `glfw3.h`는 `.cpp` 안에서만 `#include`.
- 사용자(Core/Application)는 `Window` 클래스만 보고, GLFW의 존재를 모른다.

### 4.2 입력 처리 — 콜백 vs 폴링
GLFW는 두 가지 입력 패턴을 지원. 학습 단계에선 **폴링(polling)** 권장 (직관적):

```cpp
// 폴링 (권장): 메인 루프에서 매 프레임 확인
if (window.isKeyPressed(GLFW_KEY_ESCAPE)) { /* 종료 */ }

// 콜백 (나중에): 이벤트 발생 시 함수 호출
glfwSetKeyCallback(m_window, [](GLFWwindow*, int key, int, int action, int) {
    if (key == GLFW_KEY_ESCAPE && action == GLFW_PRESS) { /* ... */ }
});
```

초기엔 폴링으로 시작하고, 입력 시스템이 커지면 이벤트 큐 + 콜백으로 발전.

## 5. glad의 위치 — RHI/OpenGL 백엔드 내부

glad는 OpenGL 함수 로더이므로 **OpenGL 백엔드 구현체 안에만** 존재해야 한다.

### 5.1 초기화 위치 — 가장 중요한 결정
`gladLoadGL()`은 **GL 컨텍스트 생성 직후**에 호출되어야 함 (04_문서 참고). 이 호출을 어디에 둘 것인가?

**권장 위치: `RHI/OpenGL/GLContext.cpp`**

```cpp
// sources/RHI/OpenGL/GLContext.h
#pragma once
struct GLFWwindow;            // 전방 선언

namespace kuma::gl {

class GLContext {
public:
    // GL 컨텍스트가 이미 생성된 GLFW 창을 받아 glad를 초기화
    static void init(GLFWwindow* window);
};

} // namespace kuma::gl
```

```cpp
// sources/RHI/OpenGL/GLContext.cpp
#include "GLContext.h"
#define GLFW_INCLUDE_NONE
#include <GLFW/glfw3.h>
#define GLAD_GL_IMPLEMENTATION
#include <glad/gl.h>           // glad는 여기서만 include

namespace kuma::gl {

void GLContext::init(GLFWwindow* window) {
    glfwMakeContextCurrent(window);
    int version = gladLoadGL(glfwGetProcAddress);
    if (version == 0) { /* 에러: glad 로드 실패 */ }
}

} // namespace kuma::gl
```

### 5.2 `#define GLAD_GL_IMPLEMENTATION`의 위치 ⚠️
glad2에서 구현체를 활성화하는 매크로는 **정확히 하나의 .cpp 파일에서만** 정의되어야 한다.
→ `RHI/OpenGL/GLContext.cpp`가 그 위치.

> 다른 GL 백엔드 파일(`GLBuffer.cpp`, `GLShader.cpp`)에서는 `glad/gl.h`만 include하고, 이 매크로는 정의하지 않음.

### 5.3 의존성 경계 시각화
```
Platform/Window.cpp ──── #include <GLFW/glfw3.h>        ✅ GLFW는 여기
                         (glfw3.h)
                              │
                              │ GLFWwindow* 전달
                              ▼
RHI/OpenGL/GLContext.cpp ─ #include <glad/gl.h>         ✅ glad는 여기
                            #include <GLFW/glfw3.h>      (glfwGetProcAddress 위해)
                              │
                              │ gl* 함수 호출
                              ▼
RHI/OpenGL/GLBuffer.cpp ── #include <glad/gl.h>         ✅ GL 구현은 여기서만
                            glBufferData(...)           Renderer/Core는 gl* 모른다
```

## 6. 의존성 매트릭스 (컴파일 타임)

이 표를 보면서 코드를 작성하면 분리가 자동으로 됨.

| 파일 | 허용되는 include | 금지되는 include |
|------|------------------|------------------|
| `Core/Application.cpp` | Window.h, Renderer.h | ❌ glfw3.h, glad/gl.h |
| `Platform/Window.cpp` | glfw3.h | ❌ glad/gl.h |
| `Renderer/Renderer.cpp` | RHI/*.h, Math/* | ❌ glfw3.h, glad/gl.h |
| `RHI/OpenGL/GLContext.cpp` | glad/gl.h, glfw3.h | — |
| `RHI/OpenGL/GLBuffer.cpp` | glad/gl.h | ❌ glfw3.h (일반적으로) |
| `RHI/Buffer.h` (추상) | 표준 헤더만 | ❌ 모든 그래픽스 헤더 |

**확인 방법**: 어떤 파일에서 `gl*` 함수를 직접 호출하면, 그 파일은 `RHI/OpenGL/` 안에 있어야 한다. 이 규칙만 지키면 됨.

## 7. 가벼운 시작을 위한 최소 구조 (지금 만들 것)

학습 단계에 맞춰, **처음 3개 클래스**만으로 시작:

```
sources/
├── Core/
│   └── Application.{h,cpp}     ← Window을 소유, 메인 루프
├── Platform/
│   └── Window.{h,cpp}          ← GLFW 래핑
├── RHI/
│   └── OpenGL/
│       └── GLContext.{h,cpp}   ← glad 초기화
└── main.cpp                    ← Application 생성 후 run
```

### 7.1 Application 클래스 — 메인 루프의 집중
```cpp
// sources/Core/Application.h
#pragma once
#include "Platform/Window.h"    // Window 타입 필요

namespace kuma {

class Application {
public:
    Application();
    void run();                 // 메인 루프
private:
    Window m_window;
};

} // namespace kuma
```

```cpp
// sources/Core/Application.cpp
#include "Application.h"
#include "RHI/OpenGL/GLContext.h"  // GL 초기화

namespace kuma {

Application::Application()
    : m_window(800, 600, "Kuma Engine") {
    gl::GLContext::init(m_window.nativeHandle());  // glad 로드
}

void Application::run() {
    while (!m_window.shouldClose()) {
        // 렌더링 (지금은 단색 클리어만)
        m_window.swapBuffers();
        m_window.pollEvents();
    }
}

} // namespace kuma
```

```cpp
// main.cpp
#include "Core/Application.h"

int main() {
    kuma::Application app;
    app.run();
    return 0;
}
```

### 7.2 의존성 흐름 (main → Application → Window + GLContext)
```
main.cpp
  → Application (Core)
      → Window (Platform)        ← GLFW
      → GLContext (RHI/OpenGL)   ← glad
```
상위 계층은 하위를 알고, 하위는 상위를 모른다. 이 흐름이 전부입니다.

## 8. 학습 단계별 확장 로드맵

| 학습 스텝 | 추가할 코드 | 들어갈 폴더 |
|-----------|-------------|-------------|
| Hello Window | Application, Window, GLContext | Core/, Platform/, RHI/OpenGL/ |
| Hello Triangle | Buffer(VertexArray), Shader | RHI/OpenGL/ |
| Textures | Texture | RHI/OpenGL/ |
| Transformations | Math (vec/mat) | Math/ |
| Camera | Camera | Renderer/ |
| Mesh/Material | Mesh, Material | Renderer/ |
| **RHI 추상화 도입 시점** | `RHI/IBuffer.h` 인터페이스 + GL 구현 | RHI/ (루트에 인터페이스) |
| 두 번째 백엔드 (Vulkan/D3D) | VulkanBackend | RHI/Vulkan/ |

> 💡 **추상화 도입 타이밍**: GL 백엔드가 Buffer/Shader/Texture를 갖추고, 코드가 어느 정도 안정되었을 때 (대략 Hello Triangle → Camera 사이) 인터페이스 도입 검토. 너무 일찍 도입하면 인터페이스 설계가 계속 바뀜.

## 9. namespace 구성 제안

의존성과 더불어 네임스페이스도 계층을 반영:

```cpp
namespace kuma {            // 최상위
    class Application;      // Core

namespace platform {        // Platform 계층
    class Window;
    class Input;
}

namespace rhi {             // RHI 추상 (나중에 인터페이스)
    class IBuffer;
}

namespace gl {              // OpenGL 백엔드 구현
    class GLContext;
    class GLBuffer;         // rhi::IBuffer 구현
}

namespace renderer {        // 고수준 렌더러
    class Camera;
    class Mesh;
}
} // namespace kuma
```

`gl` 네임스페이스가 백엔드 구현을 캡슐화 → 나중에 `vk`(Vulkan), `d3d`(DirectX) 추가 시 깔끔.

## 5. 결론

### 권장 구조 요약
1. **GLFW** → `Platform/` 계층. `Window` 클래스가 래핑. 헤더에 전방 선언, cpp에만 include.
2. **glad** → `RHI/OpenGL/` 백엔드 내부. `GLContext.cpp`에 `#define GLAD_GL_IMPLEMENTATION` + `gladLoadGL`.
3. **의존성 방향**: 상위 → 하위로만. `gl*` 호출은 `RHI/OpenGL/` 안에서만.
4. **지금 만들 것**: Application(Core) + Window(Platform) + GLContext(RHI/OpenGL) 3개 클래스.
5. **추상화는 나중에**: 두 번째 백엔드 추가 시점에 인터페이스 도입.

### 시작 체크리스트
- [ ] `sources/` 하위에 Core/, Platform/, RHI/OpenGL/ 폴더 생성
- [ ] `Window.h/cpp` — GLFW 래핑 (헤더 전방 선언, cpp에 include)
- [ ] `GLContext.h/cpp` — glad 초기화 (`#define GLAD_GL_IMPLEMENTATION` 한 곳)
- [ ] `Application.h/cpp` — Window 소유, 메인 루프
- [ ] `main.cpp` — Application 생성 후 run()
- [ ] LearnOpenGL "Hello Window" 단색 출력까지 도달
